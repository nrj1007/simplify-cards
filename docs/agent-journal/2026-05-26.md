## 2026-05-28 Dev Server Addendum

- Investigated why local dev suddenly felt worse even though the app had seemed stable before.
- Confirmed the user had likely been starting the app with:
  - `.\scripts\dev.ps1`
- That script currently runs plain:
  - `next dev`
- However, on the current machine/runtime state:
  - `next dev` resolves to Turbopack
  - Turbopack is not supported on this `win32/arm64` setup because native bindings are unavailable
- Conclusion:
  - the user was right that they were not consciously using Webpack earlier
  - but the current clean-start path now requires `next dev --webpack`
- Also confirmed a second overlapping issue:
  - the Webpack/WASM dev server can fall into a corrupted `.next` state after repeated rebuilds
  - symptoms include:
    - `ENOENT` for generated files like `.next\dev\server\app\cards\[id]\page.js`
    - missing vendor chunk files like `.next\dev\server\vendor-chunks\lucide-react.js`
    - routes oscillating between `200`, `404`, and `500`
- Practical recovery flow on this machine:
  1. stop the process holding port `3000`
  2. delete `.next`
  3. restart with `npm run dev -- --webpack`
- Verified after clean restart:
  - `/`
  - `/cards/hsbc-travelone`
  - `/cards/apollo-sbi-select`
  all returned `200`
