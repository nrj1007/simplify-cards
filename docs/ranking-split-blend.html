Ranking logic: Split Blend vs Non-Split Blend

How the /recommend engine ranks & presents cards in the two modes — and where the active split-blend path has gaps. Code refs: lib/recommend.ts, lib/ranking-strategies.ts, lib/result-strategies.ts

TL;DR — "blend" means two different things

The word blend covers two separate things:

  • The ranking blend (ranking-strategies.ts): a single strategy, absolute-blend — a weighted average of each card's fitScore across the broad spend levels, producing the global ranking key normalizedFitScore. (The old alternative max-yield strategy and the blendMode switch have been removed — there is now only one ranking blend.)
  • The presentation axis — resultStrategy (result-strategies.ts): "single-list" (default) vs "reward-type-split" (Cashback / Rewards sections).

Split blend = resultStrategy === "reward-type-split" (with the always-on weighted-average ranking). Defined as isSplitBlend in recommend.ts. Non-split blend = same ranking, shown as one flat list.

The app's active default is non-split (single-list). Split is opt-in via the "group by type" toggle on /recommend (RecommendCalculator.tsx) which sends resultStrategy:"reward-type-split" to POST /api/recommend.

3. Why the cashback-fairness mechanism exists

Cashback cards usually carry monthly caps. Scored at the non-split blend's high spend levels (300k, 600k annual), they blow past their caps, so their marginal net value collapses and they rank below points/miles cards that keep earning.

The engine now re-bases every primary cashback card onto [100k,200k,300k,500k] w [1.3,1.2,1.1,1] in the global blend (isCashbackBlendCard in recommend.ts) — spend levels where caps haven't yet dominated — so the high [3L,10L,20L,30L] reward-card levels apply to reward cards only. A strong sub-₹3L cashback card is now judged fairly in the flat single-list and the split Cashback section. Rewards cards keep their high-spend evaluation, where they shine.

Remaining nuance: the split section additionally orders cashback cards by splitOrderScore; the displayed estimatedNetValue is still the card's representative-level value, so the shown figure can differ slightly from the rank-driving score.

4. Gaps in split blend vs non-split blend

Verified against current code. Severity reflects user-visible impact, not effort.

1

splitOrderScore is envelope-only

Latent
Where recommend.ts:699 — if (!useEnvelopeScoring) return scoreCardForSpend(card, spend);
Gap The single-spend path returns early and never sets splitOrderScore. A split query that takes that path would have its sections silently fall back to estimatedNetValue ordering — i.e. the split's whole reason for existing (cashback fairness) is skipped. useEnvelopeScoring is force-enabled for split views (~recommend.ts:334), so this is currently latent, but it relies on a flag staying in sync with the split branch.
Fix Either compute splitOrderScore on the non-envelope branch too, or add an invariant/assert that isSplitBlend ⇒ useEnvelopeScoring so the coupling can't silently break.
2

Three different sort keys float across the two modes

Confusing
Where candidate sort recommend.ts:879–889 (normalizedFitScore); pre-sort recommend.ts:932 (estimatedNetValue); section sort result-strategies.ts:76–97 (splitOrderScore).
Gap Toggling split changes both the partition and the sort basis. A card ranked high on net value can drop in the split section (and vice-versa), so the same query reorders in ways that look arbitrary to a user flipping the toggle.
Fix Decide and document one of: (a) split is display-only by design (then make the flat list and split agree on a single key), or (b) feed the reward-type-aware levels into the global key too so both views are consistent.
3

Cashback fairness never reaches the global rank / flat list

Fixed ✓
Was The realistic cashback spend levels were applied only inside the split section (and to the global key only for dedicated restrictToCashbackCards queries). In a mixed non-split flat list, cashback cards were ranked on the broad [3L,10L,20L,30L] levels and under-weighted by their caps.
Done The trigger was widened from isCashbackQueryBlend (cashback-restricted only) to isCashbackBlendCard (every primary cashback card). Now cashback cards are re-based to [100k,200k,300k,500k] w [1.3,1.2,1.1,1] in the global blend, so the high [3L,10L,20L,30L] levels apply to reward cards only — fair ranking in both the flat list and the split view. recommend.ts:752,761,770
4

Dual-bucket re-valued scores carry no splitOrderScore

Inconsistent order
Where reValuedScore() recommend.ts:838–852 builds rewardBucketScore/cashbackBucketScore but never sets splitOrderScore on them. Sort guard at result-strategies.ts:80–84 only compares when both are defined.
Gap A dual-bucket card shown in its secondary section is sorted by the net-value fallback while native cards there sort by splitOrderScore — a mixed ordering basis inside one section, so the dual card can land in an inconsistent slot.
Fix Compute and attach a splitOrderScore on the re-valued scores (using the same reward-type-aware levels for the destination bucket) so every card in a section sorts on the same key.
5

Pre-sort by net value is overridden when blended

Minor
Where applyResultStrategy sorts byNetValue (recommend.ts:932), then group() re-sorts by splitOrderScore when isBlend (result-strategies.ts:76).
Gap The pre-sort is discarded for ordering in split mode (it only survives as the tie-break fallback). Harmless but easy to misread as "the order is net value" when it isn't.
Fix Document the two-stage sort, or skip the pre-sort when isBlend and rely on the section sort.
6

A card's displayed value differs between modes

Trust
Where dual-bucket re-valuation recommend.ts:838–876 + result-strategies.ts:62–74.
Gap Non-split shows a dual-bucket card's primary score; split can show a re-valued (lower or different) score in its secondary section. Same card, same query, two different net-value numbers depending on the toggle — potentially confusing given the project's "AI is never the source of truth / show real values" principle.
Fix Label the re-valued figure in the secondary section (e.g. "as a rewards card: ₹X") so the difference is explained rather than surprising.
7

Stale doc: spend levels in lib/AGENTS.md

Fixed ✓
Where lib/AGENTS.md + the comment block at recommend.ts:123–127 describe a four-tier light/mid/heavy/very-heavy blend up to Rs 30L.
Gap The code had regressed to [120000, 300000, 600000] (three levels, capped at 600k) with weights [1.5, 1.25, 1] — out of sync with its own documented 3L/10L/20L/30L design, so super-premium cards that only pull ahead at very high spend were capped out.
Done Restored the blend to its documented design: spendLevels [300000, 1000000, 2000000, 3000000] (3L/10L/20L/30L ≈ Rs 25k/83k/167k/250k per month) with high-spend-leaning weights [1, 1.25, 1.5, 1.75], in ranking-strategies.ts (both strategies), the mirror consts in recommend.ts, and lib/AGENTS.md.
